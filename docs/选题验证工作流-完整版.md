# 选题验证工作流 - 完整版

> 创建时间：2026-02-02
> 目的：确保选题真实、准确、有价值，避免误导读者

---

## 🎯 核心原则

### 三大铁律

1. **真实性优先** - 宁可慢一步，不可发假新闻
2. **深度理解** - 不能只看标题和吐槽，必须理解产品本质
3. **正向价值** - 帮助读者解决问题，而非制造焦虑

### 两个禁区

❌ **禁区 1：基于单一传闻的"揭露"类选题**
- Reddit 某个用户的吐槽
- 未经证实的"泄露"
- 没有官方确认的"发布"

❌ **禁区 2：片面信息的"真相"类选题**
- 只看到问题一面，未了解完整功能
- 个例当作普遍问题
- 配置问题当作产品问题

## 📋 完整工作流

### 阶段 1️⃣：热点采集

**工具：** topic-collector

**输出：** 原始热点列表（20-30条）

**要求：**
- 多源并行采集
- 保留原文链接
- 标注来源类型（官方/媒体/社区）

---

### 阶段 2️⃣：第一轮验证 - 信息真实性

**目标：** 排除假新闻、传闻、过期信息

#### ✅ 验证检查清单

**A. 来源可信度检查**

| 来源类型 | 可信度 | 要求 | 示例 |
|---------|--------|------|------|
| **官方公告** | ⭐⭐⭐⭐⭐ | 直接采用 | Anthropic 官网、OpenAI 博客 |
| **权威媒体** | ⭐⭐⭐⭐ | 需交叉验证 | TechCrunch, The Verge, Wired |
| **技术博客** | ⭐⭐⭐ | 需官方确认 | Medium 认证作者、Dev.to |
| **社区讨论** | ⭐⭐ | 仅作线索 | Reddit, HN (需进一步验证) |
| **个人吐槽** | ⭐ | 不可直接用 | 单个用户评论 |

**B. 多源交叉验证**

**最低要求：**
- 官方来源：1 个官方确认 = 可用
- 非官方来源：至少 2 个独立权威媒体 = 可用
- 社区来源：至少 3 个独立讨论 + 技术证据 = 谨慎使用

**验证脚本：**
```
对于任何"发布/更新"类热点：
1. site:官方域名 关键词
2. "产品名" official announcement
3. 权威科技媒体 产品名 + 时间

对于"研究/报告"类热点：
1. 查找论文/报告原文
2. 确认研究机构官方发布
3. 查看同行评议状态
```

**C. 时效性检查**

- [ ] 确认事件发生时间
- [ ] 排除过期信息（超过 1 周的旧闻除非长尾价值高）
- [ ] 验证"即将发布"的具体日期

**D. 可信度分级**

| 等级 | 标准 | 处理方式 |
|------|------|---------|
| **A 级** | 官方确认 OR 2+ 权威媒体 | ✅ 进入下一轮 |
| **B 级** | 1 个权威媒体 + 技术证据 | ⚠️ 标注来源，谨慎使用 |
| **C 级** | 多个社区讨论 + 截图/代码 | ⚠️⚠️ 标注"未经官方证实" |
| **D 级** | 单一来源/传闻/泄露 | ❌ **直接排除** |

---

### 阶段 3️⃣：第二轮验证 - 产品/技术深度理解

**目标：** 避免片面理解、个例当普遍、配置问题当产品问题

#### ✅ 产品调研检查清单

**A. 官方文档查阅**

**必须查看：**
- [ ] 产品官方文档/README
- [ ] 功能列表和配置选项
- [ ] 定价/计费说明（如果涉及成本）
- [ ] 常见问题 FAQ

**查阅要点：**
- 产品真实能力是什么？
- 是否支持自定义配置？
- "槽点"是否因配置不当？
- 官方是否有说明/警告？

**B. 个例 vs 普遍问题判断**

**判断标准：**

| 信号 | 个例 | 普遍问题 |
|------|------|---------|
| **讨论量** | 1-2 个用户吐槽 | 大量用户反馈 |
| **官方回应** | 无官方回应 | 官方承认/修复计划 |
| **技术证据** | 仅个人描述 | 可复现的技术问题 |
| **解决方案** | 配置可解决 | 产品设计问题 |

**案例分析：**

```markdown
❌ 错误案例：OpenClaw 成本陷阱

发现：Reddit 1 个用户吐槽"月花 500 美元"
问题：
- 未查阅官方文档（发现支持自定义模型）
- 未判断个例 vs 普遍（只有 1 个用户提到）
- 未理解成本来源（用户选择了最贵模型）

正确处理：
- 查阅文档 → 发现可配置
- 改写角度 → "成本优化指南"
- 或完全放弃此选题
```

**C. 技术准确性验证**

**对于技术类选题：**
- [ ] 查看技术文档/API 文档
- [ ] 验证技术描述的准确性
- [ ] 确认功能是否真实存在
- [ ] 理解技术限制和边界

**对于"对比/评测"类选题：**
- [ ] 确认对比维度公平
- [ ] 了解各方案的配置选项
- [ ] 避免默认配置 vs 最优配置的不公平对比

**D. 风险评估**

**高风险选题特征：**
- 🚨 "揭露真相"类（容易片面）
- 🚨 "XX 骗局/陷阱"类（容易误导）
- 🚨 基于单一用户吐槽（个例风险）
- 🚨 未查阅官方文档（理解不足）

**对高风险选题的处理：**
1. 深度调研（2x 时间）
2. 改写角度（从"揭露"改为"指南"）
3. 或直接放弃（选择更安全的选题）

---

### 阶段 4️⃣：选题角度评估

**目标：** 确保角度合理、有价值、不误导

#### ✅ 角度评估检查清单

**A. 核心论点检查**

- [ ] 核心论点是否基于事实？
- [ ] 论点是否站得住脚？
- [ ] 是否考虑了反面证据？
- [ ] 结论是否过于绝对？

**B. 用户价值检查**

- [ ] 读者读完能获得什么？
- [ ] 是制造焦虑还是解决问题？
- [ ] 是否提供可操作的建议？
- [ ] 是否有正向引导？

**示例：**

| 角度类型 | 风险 | 推荐度 |
|---------|------|--------|
| **教程/指南** | 低 | ⭐⭐⭐⭐⭐ 推荐 |
| **对比/评测** | 中 | ⭐⭐⭐⭐ 需客观 |
| **趋势/分析** | 中 | ⭐⭐⭐⭐ 需数据支持 |
| **经验/复盘** | 低-中 | ⭐⭐⭐ 标注个人观点 |
| **揭露/真相** | 高 | ⭐⭐ 极度谨慎 |

**C. 标题合理性检查**

**避免的标题模式：**
- ❌ "XX 的惊天骗局"（过于绝对）
- ❌ "千万别用 XX"（缺乏nuance）
- ❌ "XX 让我损失 XX 万"（个例夸大）

**推荐的标题模式：**
- ✅ "XX 使用指南：避免常见误区"
- ✅ "XX vs YY：我的实测对比"
- ✅ "从 XX 到 YY：我的优化经验"

---

### 阶段 5️⃣：筛选 TOP 10

**工具：** topic-generator

**输入：** 已通过阶段 2-4 验证的热点

**评分维度（新增"可靠性"维度）：**

| 维度 | 权重 | 说明 |
|------|------|------|
| **可靠性** | 30% | 信息真实性 + 理解深度 |
| **用户价值** | 25% | 读者能获得什么 |
| **时效性** | 20% | 发布时间窗口 |
| **独特角度** | 15% | 差异化程度 |
| **写作难度** | 10% | 所需资源和时间 |

**可靠性评分细则：**
- 10 分：A 级来源 + 深度理解 + 低风险角度
- 7-9 分：B 级来源 + 基础理解 + 中风险角度
- 4-6 分：C 级来源 + 片面理解
- 0-3 分：D 级来源 OR 高风险角度

**通过标准：** 总分 ≥ 70 分 且 可靠性 ≥ 7 分

---

### 阶段 6️⃣：质量审核

**工具：** topic-reviewer

**重点检查：**
- 独特角度（30%）
- 用户价值（25%）
- 国内适配（20%）
- 标题吸引力（15%）
- 写作难度（10%）

**新增检查项：**
- [ ] 是否有潜在误导？
- [ ] 技术社区是否认可？
- [ ] 是否会引发负面争议？

**通过线：** 70 分

```
2. 官方文档 pricing/billing 部分
3. 官方 FAQ
```

### 工具 2：多源交叉验证

**模板：**
```
权威媒体搜索：
- site:techcrunch.com "产品名"
- site:theverge.com "产品名"
- site:wired.com "产品名"

社区讨论搜索：
- site:reddit.com "产品名" official
- site:news.ycombinator.com "产品名"
- site:twitter.com from:官方账号

技术证据搜索：
- site:github.com "产品名"
- "产品名" changelog
- "产品名" release notes
```

### 工具 3：个例 vs 普遍判断

**判断流程：**
```
1. 统计讨论量
   - Google: "问题描述" + "产品名"
   - Reddit/HN: 相关讨论数量
   - GitHub Issues: 相关 issue 数量

2. 查看官方回应
   - 是否有官方声明？
   - 是否在修复计划中？
   - 是否在已知问题列表？

3. 技术可复现性
   - 是否有复现步骤？
   - 其他用户是否遇到？
   - 是否与配置相关？

判断：
- 普遍问题：讨论多 + 官方承认 + 可复现
- 个例：讨论少 + 无官方回应 + 配置可解
```

## 🚨 常见陷阱和应对

### 陷阱 1：Reddit 泄露 = 真实发布

**案例：** Claude Sonnet 5 (2026-02-03)

**问题：**
- Reddit 讨论 ≠ 官方确认
- 日期可能是模型 ID，非发布日期
- 功能描述可能是猜测

**应对：**
1. 查找官方公告（网站、博客、Twitter）
2. 如无官方确认 → 标记为 D 级 → 排除
3. 可改写为"传闻汇总"（标注未证实）

---

### 陷阱 2：个别吐槽 = 产品问题

**案例：** OpenClaw 成本陷阱

**问题：**
- 1 个用户吐槽 ≠ 普遍问题
- 可能是配置不当
- 可能是极端使用场景

**应对：**
1. 查阅官方文档（配置选项）
2. 搜索更多用户反馈
3. 判断个例 vs 普遍
4. 如是个例 → 改写为"配置优化指南"

---

### 陷阱 3：功能描述不完整

**常见情况：**
- 只看到限制，未看到配置选项
- 只看到默认行为，未看到自定义能力
- 只看到问题，未看到官方解决方案

**应对：**
1. 必须查阅完整文档
2. 搜索 "产品名 configuration"
3. 搜索 "产品名 customization"
4. 查看 FAQ 和 Troubleshooting

---

### 陷阱 4：标题党诱惑

**诱惑：**
- "震撼发布"（未经证实）
- "惊天骗局"（过于绝对）
- "千万别用"（缺乏 nuance）

**应对：**
1. 标题必须基于事实
2. 避免绝对化表述
3. 提供正向价值
4. 符合技术社区规范
