# Stable Methods

本版本是保守整合版：尽量保留 `aigc-v1`、`humanizer-zh` 与 `baibaiAIGC` 已经验证过的方法，不主动按新的风格体系重写它们。

## 最高优先级约束

无论做哪一步，都必须先满足这些约束：

- 不新增事实、数据、文献、案例或实验结论
- 不改动专业术语、代码标识、路径、配置名、API 路径、引用标记和编号结构
- 保持核心逻辑不变
- 保持段落顺序不变
- 最终只输出正文

当不同方法发生冲突时，按以下优先级处理：

1. 事实、术语、结构保护
2. 去 AI 痕迹
3. 自然化与适度啰嗦
4. 节奏与“人写感”

## 第一层：保留 `aigc-v1` / `baibaiAIGC1` 的自然化方法

### 1. 动词短语扩展

将过于简短的动词或动词短语扩成更具过程感的表达：

- 管理 -> 开展……的管理工作 / 进行管理
- 交互 -> 进行交互 / 开展交互
- 配置 -> 进行配置
- 处理 -> 去处理……工作
- 恢复 -> 进行恢复
- 实现 -> 得以实现 / 来实现
- 提供 -> 有……功能 / 拥有……功能

### 2. 系统性词汇替换

保留这些经过实践验证的替换倾向：

| 原文 | 替换为 |
|------|--------|
| 采用 / 使用 | 运用 / 选用 / 把……当作……来使用 |
| 基于 | 鉴于 / 基于……来开展 |
| 利用 | 借助 / 运用 / 凭借 |
| 通过 | 借助 / 依靠 / 凭借 |
| 和 / 及 / 与 | 以及 |
| 并 | 并且 / 还 / 同时 |
| 其 | 它 / 其 |
| 原因 | 缘由 / 主要原因囊括…… |
| 符合 | 契合 |
| 适合 | 适宜 |
| 特点 | 特性 |
| 极大(地) | 极大程度(上) |
| 立即 | 马上 |

### 3. 括号内容整合

尽量把解释性括号揉进正文：

- ORM（对象关系映射） -> ORM 也就是对象关系映射 / 对象关系映射即 ORM
- 功能（如 ORM、Admin） -> 功能，比如 ORM、Admin 等
- 视图 (views.py) 中 -> 视图也就是 views.py 中

如果括号内容不是核心信息，且整合后极冗长，可谨慎省略，但不能损伤理解。

### 4. 句式微调

保留这些倾向：

- 合适时使用“把”字句
- 条件句可以从“若……则……”调整为更自然的“如果……就……”
- 可以适度加入“那么”“这样”“同时”等连接成分
- 可适度增加“了、的、地、所、会、可以、这个、方面、当中”等辅助词

### 5. 保留“适度啰嗦”

本版本保留这条已验证方法，但必须受控：

- 目的是打散过于凝练、过于说明书式的文风
- 允许解释性扩展
- 不能明显增字失控
- 不能变成低质口水话

## 第二层：保留 `humanizer-zh` / `baibaiAIGC2` 的去痕迹方法

### 核心原则

- 删除填充短语
- 打破公式结构
- 变化节奏
- 信任读者
- 删除“金句感”

### 重点清理的模式

#### 内容模式

- 过度强调意义、遗产、趋势
- 过度强调知名度和媒体报道
- 以空泛分析结尾的句子
- 宣传和广告式语言
- 模糊归因
- 提纲式“挑战/未来展望”段落

#### 语言模式

- 高频 AI 词：此外、至关重要、深入探讨、强调、展示、格局、复杂性等
- 系动词回避
- 否定式排比：不仅……而且……
- 三段式法则
- 刻意换词
- 虚假范围表达

#### 风格模式

- 破折号过度使用
- 粗体过度使用
- 列表项标题化
- 标签化输出

#### 交流模式

- “希望这对您有帮助”
- “当然！”
- 训练截止声明
- 讨好式回应

### 人写感不是胡乱加戏

`humanizer-zh` 有一些更偏“鲜活个性”的建议，但在论文、课程作业、技术文档场景里，要保守使用：

- 不凭空加立场
- 不凭空加“我”
- 不把严肃文本改成随笔

目标是去掉 AI 套话，不是改变文体类型。

## 第三层：保留 `baibaiAIGC3` 的节奏与收口方法

### 1. 词汇“降维”

将高度书面化、过度凝练的表达换成更日常但仍专业的说法：

- 内涵 -> 包含的意义
- 演变 -> 转变过程
- 将……定义为 -> 把……界定成

### 2. 句法冗余化重构

保留这些方法：

- 将简单定语扩成“的”字短语或从句
- 增加适量状语或后缀
- 将动词名词化并搭配新的动词

### 3. 逻辑连接松散化

将过于紧绷、过于教科书式的逻辑连接稍微放松，但不能破坏逻辑本身。

### 4. 慢速化叙事

通过适度展开、句长变化和节奏调整，让文本不再像均匀压缩的模型输出。

### 5. 冗余中的精炼

这是最终收口步骤，必须执行：

- 删除不必要的“去”
- 审慎使用泛化量词“一个”
- 删除空洞修饰词，如“一整”“相关的”
- 保证结果是“自然化的复杂”，不是“无意义的臃肿”

## 最终收口检查

交付前必须再过一遍：

- 有没有明显新增信息
- 有没有改坏术语或路径
- 有没有为了“像人”而改得很油
- 有没有连续多句结构太像
- 有没有出现“修改后”“改写后”“润色后”等标签
- 有没有明显破坏信息密度
