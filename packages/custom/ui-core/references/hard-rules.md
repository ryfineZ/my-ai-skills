# Hard Rules

## 先定方向

- 先定平台、界面类型、主任务、视觉方向，再开始实现。
- 一个界面只保一个主方向，不要混搭成“什么都有一点”。

## 设计令牌

- 颜色、间距、圆角、阴影、边框、字体层级、动效节奏都必须走 token。
- 组件文件里不要到处散落原始值；组件消费语义 token，不直接拍颜色和尺寸。

## 颜色禁区

- 没有明确品牌或设计系统依据时，不要默认回到蓝紫渐变、蓝紫玻璃、紫白 SaaS、银灰、雾绿。
- 玻璃感或 Tahoe 感也不能靠这些安全配色偷懒。

## 文案禁区

- 不要自我解释型文案。
- 不要 AI 味重的抽象句。
- 不要系统视角文案，要站在使用者视角。
- 不要抢戏但没信息价值的大标题。
- 不要像提示词残留的词组。
- 不要重复块；能合并的块直接合并。
- 不要冗余标题和冗余解释。

## 结构禁区

- 尺寸匹配内容，不要为了“显得高级”把输入框、搜索框、按钮、卡片故意做很大。
- 间距只够分组，不要空耗空间。
- 不要嵌没有意义的外框、内框、卡片套卡片。
- 主任务必须一眼成立，装饰层不能压过信息层。

## 可读性禁区

- 小字、辅助字、半透明层文字不能发灰发虚。
- 材质、渐变、背景反射都不能影响正文和状态可读性。

## 状态与反馈

- 按钮必须有按下、处理中、成功、失败、不可点的反馈闭环。
- 同层只保留必要操作；重复、近义、同权重按钮必须合并或删掉。
- 必须补齐 `hover`、`active`、`focus`、`disabled`、`loading`、`empty`、`error`。

## 动效

- 动效只服务反馈和层级切换。
- 优先 `transform` / `opacity`，不要堆无意义特效。
