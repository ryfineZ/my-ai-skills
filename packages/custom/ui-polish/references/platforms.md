# Platform Rules

## Web

- 首屏优先看品牌感、信息节奏和响应式。
- 营销页可以更有氛围；后台和工具页更重清晰与效率。
- 页面动效可以稍强，但不能影响可读性和性能。

## H5 / Mobile Web

- 优先考虑单手操作、首屏信息压缩和纵向滚动节奏。
- 顶部、底部吸附区要谨慎，别抢占过多可视高度。
- 字号、间距、点击热区要明显比桌面端更保守、更清晰。
- 不要把桌面布局硬压成窄屏版本，要按移动场景重排优先级。

## 小程序

- 优先遵守宿主生态的导航、列表、表单和授权交互习惯。
- 首屏要更快进入主任务，避免大段装饰性内容。
- 强调卡片分组、状态提示和主动作，不适合复杂悬浮交互。
- 视觉上可以有品牌感，但要比官网更轻、更直接。

## Mobile App

- 更强调手势、触达范围、底部操作、页面切换和长期使用舒适度。
- 优先保证主流程清晰，再做氛围和细节。
- 页面层级要更少、更直观，避免像网页一样塞太多平级模块。
- 动效应服务于导航、反馈和状态切换，不做炫技转场。

## Chrome 扩展

- Popup 空间小，优先级比“漂亮”更重要的是“秒懂”和“秒操作”。
- 顶部区域只保留一个主动作，次要操作收纳。
- 避免重动画、长滚动和复杂多栏布局。
- 信息密度可以高，但必须靠层级、留白和状态控制住拥挤感。

## Electron / Tauri / 桌面 App

- 更强调长期使用舒适度，视觉应克制、稳定、耐看。
- 优先考虑多栏布局、窗口缩放、键盘操作、选中态和面板层级。
- 不要套纯网页营销风，应更像专业工具或编辑器。
- 动效更轻，反馈更明确。

## Shared

- 所有平台都要尊重现有组件库、设计系统和平台习惯。
- 参考外部项目时，优先借鉴层级、节奏和质感，不直接照搬技术栈。
